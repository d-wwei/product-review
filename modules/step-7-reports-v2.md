### 7b. 产品使用说明书

**⚠️ 反模式警告：禁止写成"产品目录"**

说明书最常见的失败模式是写成了"屏幕上有什么"的功能清单——它只列出页面元素，
不教用户怎么用、不解释为什么有价值。这不是说明书，是产品截图的文字版。

**错误的写法**（产品目录式）：
> #### US（美股）
> - **顶部三大指数卡片**: Dow Jones / NASDAQ / S&P 500
> - **快捷工具入口**: Screeners / AutoBuy(NEW) / ETFs
> - **Featured Lists**: Top Gainers / Heat List / Short Sales / 5min Movers

**正确的写法**（用户指南式）：
> #### 查看美股行情和发现投资机会
>
> 进入 Markets → US 标签，顶部显示三大指数（Dow Jones/NASDAQ/S&P 500）的实时行情。
>
> **热门排行**：首屏下方的 Featured Lists 提供多种排行视角：
> - **Top Gainers** — 今日涨幅最大的股票，适合捕捉短期动量
> - **Heat List** — 市场热度最高的股票，反映当前资金关注方向
> - **Short Sales** — 做空量排行，可用于判断市场情绪
> - **5min Movers** — 5 分钟内波动最大的股票，适合日内交易者
>
> 点击任意排行中的股票行即可进入该股票的详情页。
>
> **向下滚动**可发现更多工具：投资主题、产业链追踪、机构持仓、财报日历、
> 股息排行、行业热力图、评级变动、宏观经济数据、Earnings Beat 等。
> 每个工具的详细说明见下方各节。

**设计原则**（参考 Stripe Docs / Notion Help / Apple User Guide）：
- **用户视角**：每个功能先说"这能帮你做什么"，再说"怎么操作"
- **任务驱动**：按"用户要完成什么"组织，不按功能列表堆砌
- **渐进式披露**：先快速上手，再逐步深入
- **完整覆盖**：每个页面的所有内容都必须写入（包括滚动后才可见的内容）
- **视觉优先**：每个步骤配带标注的手机截图
- **官方信息补充**：FAQ 和排错部分必须检索官方帮助中心，不能只靠产品体验推测

**说明书的每个功能条目必须包含以下字段**：
1. **功能名称**
2. **解决的问题**：一句话说明这个功能帮用户做什么（最重要！）
3. **使用场景**：什么情况下需要用这个功能
4. **操作路径**：从首页开始的点击路径
5. **功能说明**：包含哪些选项、如何操作
6. **配图引用**：对应的截图文件名（从 screenshot-feature-map.md 查找）

错误示例（只有路径和内容）：
  "#### Earnings Beat
   路径：Markets > US > 滚动到第一个模块
   显示 EPS 超预期排名，含股票代码、公司名、涨幅。"

正确示例（有场景和问题）：
  "#### Earnings Beat（财报超预期排名）
   **帮用户做什么**：快速找到最近财报表现超出市场预期的股票，辅助发现投资机会
   **使用场景**：财报季期间，想筛选出基本面意外向好的标的时使用
   路径：Markets > US > 滚动到第一个模块
   显示 EPS/Revenue/EBIT 超预期排名，支持按超预期幅度排序。点击 > 进入完整列表。
   配图：markets-us-earnings-beat.png"

**覆盖性保障规则**（新增，防止只写首屏）：
1. **必须读完所有探索报告**：生成说明书前，逐个 Read `exploration-reports/` 下的文件
2. **按探索报告逐项核对**：探索报告中记录的每个功能/入口都必须在说明书中出现
3. **滚动后内容同等重要**：探索报告中标注为"scroll-3 后发现"的内容不能省略
4. **功能树完整性检查**：写完后对照 exploration-queue.md 检查是否有遗漏模块

#### 7b-B：详尽模式使用说明书（MECE）

**覆盖范围**：遍历产品的每一个功能、每一个设置项、每一个交互细节。
**组织原则**：MECE（Mutually Exclusive, Collectively Exhaustive）
- **相互独立**：每个章节/功能模块边界清晰，不重叠
- **完全穷尽**：不遗漏任何功能、设置项、操作路径

**⚠️ 关键要求：完整覆盖 = 包含所有滚动后内容**

说明书最严重的失败是"只写了首屏"。例如 Markets→US 页面，首屏只有三大指数和
Featured Lists，但向下滚动还有 10+ 个功能模块（投资主题、产业链、机构追踪、
财报日历、股息排行、热力图、评级变动、宏观经济、Earnings Beat 等）。
如果只写首屏内容，相当于丢失了 60-70% 的页面功能。

**穷尽性保障流程**：
1. **读取所有探索报告**：生成前逐个 Read `exploration-reports/` 下的每个文件
2. **提取完整功能清单**：从每份探索报告中提取该模块的所有功能（包括标注为 scroll-N 后发现的）
3. **对照功能树**：写完后与 1.2 节的功能全景图逐项核对，标记遗漏
4. **交叉验证**：官方帮助中心的功能分类作为补充检查

写入 `{BASE_DIR}/manual-full.md`：

### 7d. 报告生成质量保障规则

**⚠️ 拆分是默认行为，不是 fallback。**

> **血的教训**：在实测中，"单 agent 写大型报告"反复失败了 5 次（专家评审超时 64min、
> 3 报告并行 2 个失败、UE 报告内部错误、配图说明书内部错误），每次都是失败后才手动拆分。
>
> **根因**：单 agent 的有效处理上限远低于预期。从成功/失败案例反推的安全阈值是：
> - 读取文件：≤ 3 个文件，≤ 800 行
> - 输出长度：≤ 500 行
> - Prompt 大小：≤ 3000 字
>
> 超过任一阈值就大概率失败。因此，**所有报告默认拆分生成**，除非规模确认在安全阈值内。

```
报告生成流程：

0. 【强制拆分规则】（不是"判断是否需要拆分"，而是"默认拆分，判断是否可以不拆"）：

   安全阈值（单个 agent 可处理的上限）：
   | 维度 | 安全阈值 | 超过即必须拆分 |
   |------|---------|--------------|
   | 读取文件数 | ≤ 3 个 | > 3 个文件 |
   | 读取总行数 | ≤ 800 行 | > 800 行 |
   | 预期输出 | ≤ 500 行 | > 500 行 |
   | Prompt 大小 | ≤ 3000 字 | > 3000 字 |

   只有当所有 4 个维度都在安全阈值内时，才允许单 agent 生成。
   否则必须拆分——没有例外，不要"先试一下单 agent"。

   多份报告串行生成，不在同一轮中同时写两份报告。

0.5 【固定拆分方案】（不需要判断，按报告类型直接执行对应方案）：

   ┌─────────────────┬──────────────────────────────────────────────────────┐
   │ 报告类型         │ 固定拆分方案                                          │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ expert-review   │ 按评审模块拆（通常 3-5 个 agent）                       │
   │                 │ 每个 agent 只读 1-2 个 exploration-report              │
   │                 │ 每个 agent 输出独立模块评审文件                          │
   │                 │ 主 agent cat 合并为最终文件                             │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ manual-full     │ 按一级章节拆（通常 3 个 agent）                         │
   │                 │ Part A: Watchlists + Markets（读 G1, G3, exploration-log）│
   │                 │ Part B: Stock Detail + Trading（读 G11, G23）            │
   │                 │ Part C: Accounts + Discover + Me + AI（读 G14, G20）     │
   │                 │ 主 agent cat 合并，Write 文件头                         │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ manual-full     │ 配图版：基于无图版的各 part 文件分别插图                    │
   │ (illustrated)   │ 每个 agent 读对应 part（≤800行）+ 插入截图引用            │
   │                 │ 不要让单个 agent 读完整的无图版再插图                      │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ report (UE)     │ 拆为上下两部分：                                        │
   │                 │ 上半: 执行摘要 + 整体UX + 前半模块评估                    │
   │                 │ 下半: 后半模块评估 + 分群差异 + Top10 + 评分              │
   │                 │ 每部分只读 1 个辅助文件 + prompt 内嵌关键数据              │
   │                 │ 主 agent cat 合并                                       │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ optimization    │ 单 agent，但数据全部内嵌到 prompt（≤3000字摘要）           │
   │                 │ 不让 agent 读任何文件                                    │
   │                 │ 如果摘要超 3000 字，也拆为 P0+P1 和 P2+P3 两个 agent      │
   ├─────────────────┼──────────────────────────────────────────────────────┤
   │ manual-core     │ 通常在阈值内，可单 agent                                │
   │                 │ 但 prompt 中内嵌关键数据，不让 agent 读 > 3 个文件         │
   └─────────────────┴──────────────────────────────────────────────────────┘

   合并方法统一用 Bash: cat part1.md part2.md part3.md > final.md
   合并后删除临时 part 文件。

1. 主 agent 读取 exploration-reports/ 下的所有文件（或由 subagent 分批读取）
   - 如果文件数 > 6 或总行数 > 3000，分批读取（每批 3 个文件）
   - 读取时提取关键数据点，不需要记住所有细节

2. 分段写入报告（避免单次 Write 过大导致超时）：
   a) 第一次 Write: 写入报告框架（模板标题 + 各章节标题 + 基本信息 + 目录）
      → 约 30-50 行
   b) 逐章节 Edit 追加内容：
      → 每次 Edit 追加一个章节（50-150 行）
      → 每个章节基于对应的探索报告数据
   c) 最后一次 Edit: 追加附录和截图引用
   → 总计 3-6 次写入操作，每次控制在 150 行以内

3. 报告写入后，主 agent 自验证：
   a) Read 报告文件检查是否完整
   b) 行数是否合理：
      - manual-full.md > 500 行
      - report.md > 300 行
      - optimization.md > 200 行
      - manual-core.md > 200 行
   c) 是否包含所有必需章节（按模板检查）
   d) 关键数据是否有截图依据

3.5 【事实核验】（报告发布前的最后防线）

    主 agent 必须对报告中的关键事实性声明进行核验，尤其是负面评价。
    可由主 agent 直接执行，或派生专门的"事实核验 subagent"。

    核验流程：
    a) 从报告中提取所有事实性声明（功能存在/不存在、数量、流程步骤数等）
    b) 重点核验以下类型：
       - "缺失 XX 功能" → 在模拟器上实际确认是否真的缺失（注意水平滚动、深层菜单）
       - "需要 N 步才能到达" → 实际数一遍步骤数
       - "XX 功能不存在" → 检查是否在其他入口存在
       - Persona 声称的具体痛点 → 核实底层探索数据是否支持
    c) 核验方式（按成本从低到高选择）：
       - 查阅 exploration-reports/：检查探索报告中是否有该功能的记录
       - 查阅 screenshots/：截图中是否有视觉证据
       - 实际操作验证：在模拟器上重新执行相关操作（仅对高影响声明）
    d) 核验结果处理：
       - 确认属实 → 保留
       - 发现不准确 → 直接 Edit 修正报告内容
       - 无法确认 → 在报告中标注 "[待确认]"，不做确定性陈述

    必须核验的高优先级声明：
    - 所有 "缺失/不存在/没有" 类断言
    - 所有被用作 P0 级建议依据的事实
    - 所有 Persona 评分最低项的理由
    - 所有涉及合规风险的声明

    实测教训：moomoo 评测中，"Me Tab 需双击才能进入主菜单" 和 "期权链缺少 Greeks"
    两个核心负面发现均为事实性错误——前者是 agent 的错误归纳，后者是因为未执行水平滚动。
    这两个错误如果未被核验就发布，会严重损害报告公信力。

4. ⚠️ 覆盖完整性验证（manual 模式必须执行）：
   a) 重新 Read 每个 exploration-reports/ 文件
   b) 从中提取所有功能名称/模块名称
   c) 在报告中 Grep 这些名称，确认每个都出现
   d) 未出现的 → 直接 Edit 补充到对应章节
   e) 特别关注：探索报告中标注为 "scroll-N 后发现" 的功能
      → 这些最容易被遗漏，必须逐一核对

5. 验证失败时：直接 Edit 修补缺失章节（不重写整个文件）

6. 报告数据一致性检查：
   - 报告中引用的数据必须来自探索数据（如"支持8种订单类型"
     必须与 exploration-reports/ 中实际记录的一致）
   - 不可编造未探索到的功能或数据

7. 写作风格检查（manual 模式）：
   - 抽查 3 个章节，确认不是纯"产品目录"式列举
   - 每个功能必须有"做什么+怎么用"两部分，缺任一 → Edit 补充
   - 禁止出现纯粹的 "- 功能A\n- 功能B\n- 功能C" 列表（无解释）

多报告处理规则：
- 用户请求多份报告时（如 manual + ue-report + optimization + expert-review），
  每份独立完成（写入+验证）后再开始下一份
- 大型报告（expert-review、manual-full）拆分为多个 subagent 分段生成后合并
- subagent 的 prompt 不超过 8000 字，每个 subagent 的输出不超过 200 行
- 主 agent 负责合并和质量验证
```
