#### 7a 拆分策略（ue-report 必须遵循）

15 章体验报告**不可能由单个 agent 一次性完成**。必须按以下策略拆分为多个 agent 串行执行。

**Step 1: 主 agent 准备数据摘要**

在派发 subagent 之前，主 agent 先读取所有源数据（exploration-log、persona-feedback、expert-review 等），
生成一个**精简数据摘要文件**（≤100 行），包含：产品基本信息、结构概要、Persona 评分、
核心数据点、专家评审要点、竞品摘要。所有 subagent 只读这一个文件。

**数据摘要一致性校验**（防止 subagent 从同一文件取到矛盾数据）：
- Persona 评分只写一套（取 persona-feedback-synthesis 的权威分数），附带均值
- 验证：均值 = 各分数之和 ÷ 人数，手动计算确认，不允许四舍五入误差 > 0.1
- 数据点不写两种表述（如不要头部写"均值6.2"正文又写整数分均值5.75）
- 关键发现用统一措辞（如 Greeks 易发现性问题只用一种标准表述，不要一处写"缺失"另一处写"需左滑"）

**Step 2: 按内容复杂度分组**

15 章分为 3 类内容，每类的 agent 承载能力不同：

| 类别 | 章节 | 特征 | 单 agent 承载上限 |
|------|------|------|------------------|
| **描述型** | 四~六（结构/框架/表现）、十一（性能） | ASCII 图表、配色表、固定模式重复 | 可合并，≤500 行 |
| **分析型** | 七（模块深度评估） | N 模块 × 6 维度交叉推理，每个模块数据源不同 | **≤3 个异构模块/agent** |
| **合成型** | 一~三（摘要/战略/范围）、八~十（旅程/分群/AI 专项）、十二~十五（亮点/改进/竞品/评分） | 从上游数据提炼洞察 | 可按 3-4 章合并，≤450 行 |

**Step 3: 推荐拆分方案**

```
Part A: 一~三（执行摘要 + 战略层 + 范围层）      → 合成型，1 agent
Part B: 四~六（结构层 + 框架层 + 表现层）         → 描述型，1 agent
Part C: 七（核心模块深度评估）                     → 分析型，拆分见下
Part D: 八~十一（体验地图 + 分群 + AI 专项 + 性能）→ 混合型，1 agent
Part E: 十二~十五（亮点 + 改进 + 竞品 + 评分）    → 合成型，1 agent
```

**Part C 的二次拆分规则**（第七章）：
- 统计核心模块数 N（通常 = 底部 Tab 数，如 6 个）
- 如果 N ≤ 3：单 agent 可完成
- 如果 N = 4~6：拆为 C1（前 3 模块）+ C2（后 N-3 模块）
- 如果 N > 6：每 2~3 个模块一个 agent
- **每个模块必须包含全部 6 个维度**（概述/体验/Persona 差异/专家洞察/建议/截图参考）

**边缘节显式分配**（防止报告头/附录在拆分中被遗漏）：
- **报告标题 + 基本信息块**：分配给 Part A，作为文件开头
- **附录**：分配给 Part E，作为文件末尾
- 拆分 prompt 中必须明确写出"文件以附录结束"，否则 agent 会在最后一章结尾后直接停止

**跨 agent 去重协调**（防止同一发现被 5 个 agent 各写一遍）：
- 在数据摘要中标注 **Top 3 核心发现的"权威章节"**，例如：
  - "Greeks 易发现性 → 权威位置: 7.2 期权模块，其他章节引用时写'详见 7.2 节'"
  - "TFSA Room 缺失 → 权威位置: 7.4 Accounts，其他章节简述+交叉引用"
- 执行摘要（第一章）允许重述核心发现（摘要的天然职责），但限定在 1-2 句以内
- 其他章节遇到已标注的核心发现时，只写一句结论 + "（详见 X.X 节）"
- **反模式**：Part D 和 Part E 不应再完整描述"David 因 Greeks 缺失给出最低分"的完整故事

**Step 4: 串行执行 + 合并**

1. 按 Part A → B → C1 → C2 → D → E 串行分派（一次只派 1 个 agent）
2. 每个 agent 完成后立即 `wc -l` 验证行数
3. 全部完成后 `cat` 合并为 report.md
4. **合并后清理**：删除 Part 标题（如 "# ... Part B"）和尾注（如 "*Part X 完*"）
5. **合并后校验**：只有 1 个一级标题 + 15 个二级标题连续 + 无重复章节号

**Step 5: 合并后质量审计（必选）**

合并后主 agent 执行以下 3 项检查，发现问题直接用 Edit 修复，无需重新派发 agent：

**5a. 数值一致性审计**：
- `grep` 所有 Persona 评分引用，确认与数据摘要一致
- `grep` 所有均值引用，确认计算正确且全文统一
- 检查加权评分表的算术（权重之和 = 100%，加权分 = 评分 × 权重）

**5b. 前版对比审计**（如存在前版报告）：
- Read 前版的章节目录，逐章比对 V4 是否覆盖
- 重点关注前版中有但 V4 数据摘要中未收录的细节（如 bug 发现、边缘功能、差异化特性）
- 缺失内容直接 Edit 补入对应章节

**5c. 去重审计**：
- `grep` Top 3 核心发现的关键词（如 "Greeks""TFSA Room""Onboarding"），检查出现次数
- 如某发现完整描述（>2 句）出现超过 2 处（执行摘要 + 权威章节），将多余处缩减为交叉引用

### 7b. 产品使用说明书

**⚠️ 反模式警告：禁止写成"产品目录"**

说明书最常见的失败模式是写成了"屏幕上有什么"的功能清单——它只列出页面元素，
不教用户怎么用、不解释为什么有价值。这不是说明书，是产品截图的文字版。

**错误的写法**（产品目录式）：
> #### US（美股）
> - **顶部三大指数卡片**: Dow Jones / NASDAQ / S&P 500
> - **快捷工具入口**: Screeners / AutoBuy(NEW) / ETFs
> - **Featured Lists**: Top Gainers / Heat List / Short Sales / 5min Movers

**正确的写法**（用户指南式）：
> #### 查看美股行情和发现投资机会
>
> 进入 Markets → US 标签，顶部显示三大指数（Dow Jones/NASDAQ/S&P 500）的实时行情。
>
> **热门排行**：首屏下方的 Featured Lists 提供多种排行视角：
> - **Top Gainers** — 今日涨幅最大的股票，适合捕捉短期动量
> - **Heat List** — 市场热度最高的股票，反映当前资金关注方向
> - **Short Sales** — 做空量排行，可用于判断市场情绪
> - **5min Movers** — 5 分钟内波动最大的股票，适合日内交易者
>
> 点击任意排行中的股票行即可进入该股票的详情页。
>
> **向下滚动**可发现更多工具：投资主题、产业链追踪、机构持仓、财报日历、
> 股息排行、行业热力图、评级变动、宏观经济数据、Earnings Beat 等。
> 每个工具的详细说明见下方各节。

**设计原则**（参考 Stripe Docs / Notion Help / Apple User Guide）：
- **用户视角**：每个功能先说"这能帮你做什么"，再说"怎么操作"
- **任务驱动**：按"用户要完成什么"组织，不按功能列表堆砌
- **渐进式披露**：先快速上手，再逐步深入
- **完整覆盖**：每个页面的所有内容都必须写入（包括滚动后才可见的内容）
- **视觉优先**：每个步骤配带标注的手机截图
- **官方信息补充**：FAQ 和排错部分必须检索官方帮助中心，不能只靠产品体验推测

**说明书的每个功能条目必须包含以下字段**：
1. **功能名称**
2. **解决的问题**：一句话说明这个功能帮用户做什么（最重要！）
3. **使用场景**：什么情况下需要用这个功能
4. **操作路径**：从首页开始的点击路径
5. **功能说明**：包含哪些选项、如何操作
6. **配图引用**：对应的截图文件名（从 screenshot-feature-map.md 查找）

错误示例（只有路径和内容）：
  "#### Earnings Beat
   路径：Markets > US > 滚动到第一个模块
   显示 EPS 超预期排名，含股票代码、公司名、涨幅。"

正确示例（有场景和问题）：
  "#### Earnings Beat（财报超预期排名）
   **帮用户做什么**：快速找到最近财报表现超出市场预期的股票，辅助发现投资机会
   **使用场景**：财报季期间，想筛选出基本面意外向好的标的时使用
   路径：Markets > US > 滚动到第一个模块
   显示 EPS/Revenue/EBIT 超预期排名，支持按超预期幅度排序。点击 > 进入完整列表。
   配图：markets-us-earnings-beat.png"

**覆盖性保障规则**（新增，防止只写首屏）：
1. **必须读完所有探索报告**：生成说明书前，逐个 Read `exploration-reports/` 下的文件
2. **按探索报告逐项核对**：探索报告中记录的每个功能/入口都必须在说明书中出现
3. **滚动后内容同等重要**：探索报告中标注为"scroll-3 后发现"的内容不能省略
4. **功能树完整性检查**：写完后对照 exploration-queue.md 检查是否有遗漏模块

#### 7b-A：Core 模式使用说明书

**定位**：Core 版不是"精简版操作手册"，而是**产品价值指南**。
它的核心使命是让读者快速理解：
1. **为什么选这个产品**（竞争优势，有数据支撑）
2. **怎么用好它最核心的能力**（操作路径）
3. **怎么快速上手**（速查）

**覆盖范围**：核心功能 + 竞争优势 + 最常用场景。
**筛选标准**：
- 产品的核心价值流程（没有这个功能产品就不成立）
- 独特/差异化功能（竞品没有的）
- 高频使用功能（用户每天都会用的）
- 新手最可能卡住的地方

**⚠️ 反模式警告：禁止写成"精简版操作手册"**

Core 版最常见的失败模式是写成了详尽版的缩略版——只保留操作步骤，
删掉了所有深度内容。这样的文档把产品名换成任何竞品都能成立，
读完后用户不知道"为什么选这个产品"。

错误示例（纯操作手册式）：
> **Markets**
> 什么时候用：想看市场行情时
> 1. 点击 Markets Tab → 2. 选择市场 → 3. 浏览排行榜 → 4. 向下滚动

正确示例（价值指南式）：
> **用 Markets 发现投资机会**
> 什么时候用：想了解市场全貌、寻找投资灵感时
> moomoo 的优势：单个市场页面集成 18 个可定制模块（竞品通常 3-5 个），
> 涵盖排行榜、产业链、机构追踪、热力图等专业工具，
> 且支持拖拽排序定制你的信息面板——这在移动端券商 App 中是独有的。
> 操作步骤：...

**⚠️ 反模式警告 2：禁止把"功能数量"当"竞争优势"**

写竞争优势时，不要罗列一个功能的全部子项。先把子项分为"行业标配"和"差异化项"，
只拿差异化项论证优势，并说清竞品缺失这些能力对用户意味着什么。

错误示例（功能清单式）：
> moomoo 的优势：支持 9 种订单类型（Limit / Market / Stop Limit / Stop /
> LIT / MIT / Trailing Stop Limit / Trailing Stop / Odd Lot）。

问题：Limit 和 Market 每个券商都有，把它们和 LIT/MIT 并列，用普通项稀释了稀缺项。
读者看到的是"很多类型"，但不知道"哪些是别人没有的"。

正确示例（差异化拆解式）：
> moomoo 的优势：**LIT（触价限价单）和 MIT（触价市价单）**让你设置
> "股价触及 X 时自动以 Y 价格挂单"的条件策略——这在 Wealthsimple、
> Questrade、Robinhood 上都没有。**Overnight 隔夜交易时段**让你在重大
> 新闻发布后立即反应，而竞品用户必须等到次日开盘。

方法：功能有 N 个子项时，先问"哪些是竞品也有的（标配）？哪些是竞品没有的（差异化）？"
只拿差异化项写优势，标配项可以不提或一笔带过。

#### 7b-B：详尽模式使用说明书（MECE）

**覆盖范围**：遍历产品的每一个功能、每一个设置项、每一个交互细节。
**组织原则**：MECE（Mutually Exclusive, Collectively Exhaustive）
- **相互独立**：每个章节/功能模块边界清晰，不重叠
- **完全穷尽**：不遗漏任何功能、设置项、操作路径

**⚠️⚠️ 总-分结构强制规则（v3 新增，必须遵守）**

说明书必须采用**"总-分"结构**：先给出产品全局概述，再逐模块展开。
这是说明书最基本的组织原则——读者在深入任何具体模块之前，必须先对产品整体有认知。

1. **"总"部分（一~三章）— 主 agent 必须亲自写，禁止交给 subagent**
   - **一、产品概述**：产品定位、核心价值、目标用户、整体架构、功能全景图
   - **二、下载安装与注册**（如适用）
   - **三、主界面与导航**：底部导航栏全貌 + 各 Tab 关系 + 全局功能
   - ⏱ **写作时机**：在拆分任何模块 subagent **之前**先完成"总"部分

2. **"分"部分（四章起）— 可拆分给 subagent**
   - 每个一级模块一个独立章节（四、五、六…）
   - subagent 只写被分配的模块内容，不加全局标题页

3. **合并校验**：主 agent 收到所有 subagent 输出后，必须检查：
   - ✅ 一~三章完整存在且有实质内容（非空壳）
   - ✅ 每个一级模块有独立章节，编号连续
   - ✅ 无 "Part X" 分割标记残留
   - ✅ 全文只有一个 `# {App 名称} 完整使用说明书` 一级标题

**⚠️⚠️ 一级模块划分原则（v3 新增）**

- App **底部导航栏的每个 Tab** = 一个独立的一级章节（四、五、六…）
- 它们是**并列关系**，层级相同，**严禁合并**到同一章节
- 章节编号应与 Tab 在导航栏中的排列顺序一致
- 不属于任何 Tab 的全局功能（如搜索、AI 助手、通知中心）归入独立章节
- subagent 分派时：**每个 Tab 至少独占一个 subagent**，禁止把两个 Tab 塞进同一个 subagent

错误示例（两个并列 Tab 被合并成一个标题）：
> `# moomoo 产品说明书 — Watchlists 与 Markets`

正确示例（每个 Tab 独立成章，层级一致）：
> `## 四、Watchlists（自选股管理）：追踪和管理你关注的股票`
> `## 五、Markets（行情中心）：浏览全球市场行情与投资工具`

**⚠️⚠️ 复杂 Tab 的二次拆分规则（v4 新增）**

并非所有 Tab 的复杂度相同。一个 subagent 能处理的内容取决于**功能节点数**，而非行数。

| Tab 复杂度 | 判断标准 | 拆分策略 |
|-----------|---------|---------|
| **简单** | 功能节点 ≤10（如 Settings、Me） | 1 个 subagent 即可 |
| **中等** | 功能节点 11-25（如 Accounts 5 子 Tab 各含 3-5 功能） | 1 个 subagent，但 prompt 中列出功能清单防遗漏 |
| **复杂** | 功能节点 >25 或含多级嵌套子 Tab（如 Markets 11 子 Tab × 各 15-20 模块） | **必须拆分** |

**复杂 Tab 的拆分方法**（以 Markets 为例）：
- 按子 Tab 分组：每 3-4 个子 Tab 一个 subagent
  - 例：subagent-M1 负责 US + CA + ETFs，subagent-M2 负责 Options + Futures + Crypto，subagent-M3 负责其余
- 每个 subagent 输出以 `### X.Y {子Tab名}` 开头，主 agent 合并到章节框架中
- 功能清单必须在 prompt 中给出（从 exploration-log 提取），防止 subagent 遗漏滚动后内容

**数据传递策略**：
- 主 agent 在拆分前，为每个 subagent 准备一份**模块功能清单**（从 exploration-reports 提取）
- 清单格式：`子Tab名 > 功能名 > 探索报告文件名 > 截图文件名`
- subagent 只需 Read 对应的 1-3 份探索报告，不需读全部
- 如探索报告总量 >20 个文件，主 agent 先构建 `exploration-summary.md`（≤150 行），subagent 读摘要 + 对应的少量原始报告

**失败恢复协议**（与 7a 通用）：
1. subagent 报错 → 先 `ls` 检查输出文件是否已部分写入
2. 已写入 → Read 检查，补写缺失部分（主 agent 直接 Edit）
3. 未写入 → 不要重试同一策略，直接将该 Tab 再拆分为更小单元
4. 连续 2 次失败 → 主 agent 亲自写该 Tab（放弃 subagent 分派）

**⚠️ 关键要求：完整覆盖 = 包含所有滚动后内容**

说明书最严重的失败是"只写了首屏"。例如 Markets→US 页面，首屏只有三大指数和
Featured Lists，但向下滚动还有 10+ 个功能模块（投资主题、产业链、机构追踪、
财报日历、股息排行、热力图、评级变动、宏观经济、Earnings Beat 等）。
如果只写首屏内容，相当于丢失了 60-70% 的页面功能。

**穷尽性保障流程**：
1. **读取所有探索报告**：生成前逐个 Read `exploration-reports/` 下的每个文件
2. **提取完整功能清单**：从每份探索报告中提取该模块的所有功能（包括标注为 scroll-N 后发现的）
3. **对照功能树**：写完后与 1.2 节的功能全景图逐项核对，标记遗漏
4. **交叉验证**：官方帮助中心的功能分类作为补充检查

### 7d. 报告生成质量保障规则

**⚠️ 拆分是默认行为，不是 fallback。**

> **血的教训**：在实测中，"单 agent 写大型报告"反复失败了 5 次（专家评审超时 64min、
> 3 报告并行 2 个失败、UE 报告内部错误、配图说明书内部错误），每次都是失败后才手动拆分。
>
> **根因**：单 agent 的有效处理上限远低于预期。从成功/失败案例反推的安全阈值是：
> - 读取文件：≤ 3 个文件，≤ 800 行
> - 输出长度：≤ 500 行
> - Prompt 大小：≤ 3000 字
>
> 超过任一阈值就大概率失败。因此，**所有报告默认拆分生成**，除非规模确认在安全阈值内。

```
报告生成流程：

0. 【强制拆分规则】（不是"判断是否需要拆分"，而是"默认拆分，判断是否可以不拆"）：

   安全阈值（单个 agent 可处理的上限）：
   | 维度 | 安全阈值 | 超过即必须拆分 |
   |------|---------|--------------|
   | 读取文件数 | ≤ 3 个 | > 3 个文件 |
   | 读取总行数 | ≤ 800 行 | > 800 行 |
   | 预期输出 | ≤ 500 行 | > 500 行 |
   | Prompt 大小 | ≤ 3000 字 | > 3000 字 |

   只有当所有 4 个维度都在安全阈值内时，才允许单 agent 生成。
   否则必须拆分——没有例外，不要"先试一下单 agent"。

   多份报告串行生成，不在同一轮中同时写两份报告。

3.5 【事实核验】（报告发布前的最后防线）

    主 agent 必须对报告中的关键事实性声明进行核验，尤其是负面评价。
    可由主 agent 直接执行，或派生专门的"事实核验 subagent"。

    核验流程：
    a) 从报告中提取所有事实性声明（功能存在/不存在、数量、流程步骤数等）
    b) 重点核验以下类型：
       - "缺失 XX 功能" → 在模拟器上实际确认是否真的缺失（注意水平滚动、深层菜单）
       - "需要 N 步才能到达" → 实际数一遍步骤数
       - "XX 功能不存在" → 检查是否在其他入口存在
       - Persona 声称的具体痛点 → 核实底层探索数据是否支持
    c) 核验方式（按成本从低到高选择）：
       - 查阅 exploration-reports/：检查探索报告中是否有该功能的记录
       - 查阅 screenshots/：截图中是否有视觉证据
       - 实际操作验证：在模拟器上重新执行相关操作（仅对高影响声明）
    d) 核验结果处理：
       - 确认属实 → 保留
       - 发现不准确 → 直接 Edit 修正报告内容
       - 无法确认 → 在报告中标注 "[待确认]"，不做确定性陈述

    必须核验的高优先级声明：
    - 所有 "缺失/不存在/没有" 类断言
    - 所有被用作 P0 级建议依据的事实
    - 所有 Persona 评分最低项的理由
    - 所有涉及合规风险的声明

    实测教训：moomoo 评测中，"Me Tab 需双击才能进入主菜单" 和 "期权链缺少 Greeks"
    两个核心负面发现均为事实性错误——前者是 agent 的错误归纳，后者是因为未执行水平滚动。
    这两个错误如果未被核验就发布，会严重损害报告公信力。

4. ⚠️ 覆盖完整性验证（manual 模式必须执行）：
   a) 重新 Read 每个 exploration-reports/ 文件
   b) 从中提取所有功能名称/模块名称
   c) 在报告中 Grep 这些名称，确认每个都出现
   d) 未出现的 → 直接 Edit 补充到对应章节
   e) 特别关注：探索报告中标注为 "scroll-N 后发现" 的功能
      → 这些最容易被遗漏，必须逐一核对

5. 验证失败时：直接 Edit 修补缺失章节（不重写整个文件）

6. 报告数据一致性检查：
   - 报告中引用的数据必须来自探索数据（如"支持8种订单类型"
     必须与 exploration-reports/ 中实际记录的一致）
   - 不可编造未探索到的功能或数据

7. 写作风格检查（manual 模式）：
   - 抽查 3 个章节，确认不是纯"产品目录"式列举
   - 每个功能必须有"做什么+怎么用"两部分，缺任一 → Edit 补充
   - 禁止出现纯粹的 "- 功能A\n- 功能B\n- 功能C" 列表（无解释）

多报告处理规则：
- 用户请求多份报告时（如 manual + ue-report + optimization + expert-review），
  每份独立完成（写入+验证）后再开始下一份
- 大型报告（expert-review、manual-full）拆分为多个 subagent 分段生成后合并
- subagent 的 prompt 不超过 8000 字，每个 subagent 的输出不超过 200 行
- 主 agent 负责合并和质量验证
```
