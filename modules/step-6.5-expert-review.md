# Step 6.5: 专家评审模式（Expert Review）

> 本文件由 SKILL.md 按需加载。当使用 `--expert-review` 标志或 `expert-review` 子命令时加载。
> 可与任何子命令组合使用（suggestion、ue-report、manual 等）。

## 为什么需要专家评审

常规 Persona 模拟产出的是**用户级反馈**——"这个功能好不好用、有没有 bug"。
这是初级 PM 水平。

**专家评审**产出的是**产品策略级洞察**——"这个模块应该增加什么功能、
应该怎么重新设计用户工作流、竞品在这个场景下做了什么、行业最佳实践是什么"。
这是高级/资深 PM + 领域专家水平。

**输出差异对比**：

| 层级 | 期权模块的输出示例 |
|------|----------------|
| 初级 PM（常规 persona） | "期权链缺少 Gamma/Vega 显示" |
| 高级 PM（专家评审） | "针对 0DTE 交易者：应增加 0DTE 异动榜单（筛选 Call/Put Volume Ratio 极高且 IV 飙升的标的）、高赔率专区（期权价格 <$1 但标的剧烈波动）、ATM 一键下单快捷入口、金额下单模式、Smart Limit 订单类型" |

---

## 触发条件

```
以下任一条件触发专家评审：
1. 用户使用 --expert-review 标志（任何子命令）
2. 用户使用 expert-review 子命令
3. 产品包含高专业度模块（如期权、衍生品、AI 工具、专业创作工具）
   且主 agent 在 Step 5 探索中识别到该模块复杂度为"高"
   → 在 Phase 3 开始前用 AskUserQuestion 建议启用
```

## 6.5.1 识别需要专家评审的模块

主 agent 在 Step 5 探索完成后，扫描 exploration-queue.md 和 exploration-reports/，
识别满足以下条件的模块：

```
需要专家评审的模块特征：
1. 高领域门槛：需要专业知识才能理解的功能（期权链、财务报表、AI 模型配置等）
2. 复杂工作流：涉及多步骤、多决策点的用户流程（交易下单、投资组合构建等）
3. 竞品差异化核心：产品的核心竞争力所在模块
4. 用户细分场景多：同一模块服务多种不同需求的用户群（如期权模块服务保守对冲者和激进日内交易者）

输出：需专家评审的模块列表 + 每个模块应由哪类专家评审
```

## 6.5.2 专家评审 Persona 生成

**专家评审 Persona 与常规 Persona 的核心区别**：

| 维度 | 常规 Persona | 专家评审 Persona |
|------|-------------|-----------------|
| 身份 | "我是一个普通用户" | "我是这个领域的资深从业者 + 产品专家" |
| 评价视角 | "这个功能好不好用" | "这个模块缺什么功能、应该怎么重新设计" |
| 知识深度 | 领域知识注入（200-300字） | **工作流场景库**（500-800字，含具体使用场景和边界情况） |
| 竞品认知 | "用过 2-3 个竞品" | **竞品功能矩阵**（精确到功能点级别的对比） |
| 输出格式 | "我觉得…因为…" | **结构化产品建议**（场景→问题→方案→预期收益→成本评估） |

### 专家 Persona 档案结构

```markdown
## Expert Reviewer #{N}: {姓名}（{一句话标签}）

### 专家身份
- **领域**: {如 "期权交易 + 量化策略"}
- **从业年限**: {如 "8年期权交易经验，管理过 $5M 期权组合"}
- **当前角色**: {如 "某对冲基金期权交易桌负责人"}
- **产品专家身份**: {如 "同时是 3 个交易平台的深度用户，曾为 Webull 提供期权功能反馈"}

### 工作流场景库（500-800 字，核心差异化内容）

该专家的日常工作流中涉及的具体场景，每个场景都是一个完整的"需求→操作→痛点"链条：

#### 场景 1: {场景名，如 "0DTE 期权日内交易"}
**我会怎么做**：{描述完整工作流，如 "每天早上 9:20 开始扫描当天到期的期权。
先看哪些股票有异常大的 Call/Put Volume Ratio（>3x），说明有方向性赌注在进场。
然后看这些股票的 IV 是否在飙升——IV 飙升意味着市场预期大波动。
找到目标后，我需要快速定位 ATM 附近的合约，看 Bid-Ask Spread 是否合理（<15% mid），
然后用 Limit Order 在 Mid Price 附近挂单。整个过程需要在 5 分钟内完成。"}

**当前产品在这个场景下的表现**：{基于探索数据评估}
**缺失/不足**：{具体功能缺失}
**竞品参照**：{竞品在这个场景下怎么做}

#### 场景 2: {下一个场景}
...（每个专家至少 3 个核心工作流场景）

### 竞品功能矩阵
| 功能点 | 本产品 | {竞品1} | {竞品2} | 行业最佳实践 |
|--------|--------|---------|---------|-------------|
| {功能} | {有/无/部分} | {描述} | {描述} | {标杆} |

### 产品创新评审维度
该专家在评审时重点关注的产品创新方向：
1. **工作流效率**: 能否减少完成核心任务的步骤/时间
2. **信息降维**: 能否把复杂信息简化为决策支持
3. **场景化设计**: 能否为不同用户场景提供定制化入口
4. **智能辅助**: 能否用算法/AI 帮用户做更好的决策
5. **差异化竞争**: 能否做出竞品没有的创新功能
```

## 6.5.3 专家评审执行协议

### 评审输入
每个专家 Persona 收到以下输入进行评审：
1. 对应模块的 exploration-reports/ 文件（完整探索数据）
2. 对应模块的截图
3. 该专家的工作流场景库
4. 该专家的竞品功能矩阵

### 评审任务

```
你是 {expert_name}，{expert_identity}。

你已经通过探索数据和截图了解了 {product_name} 的 {module_name} 模块的完整功能。

现在请你从专家角度完成以下评审：

## 任务 1: 工作流缺口分析（Workflow Gap Analysis）

对你的每个核心工作流场景，评估当前产品的支持程度：

| 工作流场景 | 当前支持度 | 关键缺口 | 影响程度 |
|-----------|----------|---------|---------|
| {场景} | 完全/部分/不支持 | {缺什么} | {对这类用户的实际影响} |

## 任务 2: 功能机会清单（Feature Opportunity List）

基于工作流缺口，提出具体的功能建议。每个建议必须包含：

### 机会 {N}: {建议标题}
**用户场景**: 什么用户在什么情况下需要这个功能
**当前痛点**: 现在用户怎么解决这个问题（workaround）
**建议方案**: 具体的功能设计描述（足够详细到开发团队能理解）
  - 入口: 在哪里提供这个功能
  - 交互: 用户怎么操作
  - 数据: 需要展示/计算什么数据
  - 降维: 如何让非专业用户也能理解（如果适用）
**竞品参照**: 有没有竞品已经做了类似功能
**预期收益**: 对用户留存/转化/满意度的影响
**实现复杂度评估**: 前端/后端/数据 各自的工作量估计（低/中/高）

## 任务 3: 现有功能优化建议

对当前已有的功能，从专家角度提出具体优化建议：

### 优化 {N}: {现有功能名} → {优化方向}
**当前实现**: {描述当前怎么做的}
**问题**: {对专业用户来说哪里不够好}
**建议**: {具体怎么改}
**优先级**: {P0/P1/P2} + 理由

## 任务 4: 用户分层设计建议（可选）

如果这个模块服务多种不同水平的用户，提出分层设计建议：

| 用户层级 | 核心需求 | 当前满足度 | 建议 |
|---------|---------|----------|------|
| 新手 | {需求} | {评分} | {建议} |
| 进阶 | {需求} | {评分} | {建议} |
| 专家 | {需求} | {评分} | {建议} |

## 任务 5: 合规风险评估（每个建议必须附带）

对任务 2（功能机会）和任务 3（优化建议）中的每条建议，评估合规风险：

- 该建议是否涉及监管敏感领域？（交易建议、跟单、AI投顾等）
- 目标市场的合规要求是什么？（如加拿大 CSA/IIROC 规定）
- 合规风险等级：低/中/高
- 如果风险为高：需要哪些前置条件（如牌照、风险披露、投资者适当性评估）

示例：
- "增加跟单功能" → 合规风险: 高
  原因：加拿大 CSA 对投资建议有严格监管，跟单可能被认定为投资建议
  前置条件：需要投资顾问牌照或豁免条件，需用户风险评估，需充分风险披露

### 合规风险评估表（每条建议必填）

| 建议 | 涉及监管领域 | 目标市场合规要求 | 风险等级 | 前置条件（高风险时） |
|------|------------|----------------|---------|------------------|
| {建议标题} | {如：投资建议/交易执行/数据隐私/无} | {具体法规} | 低/中/高 | {牌照/披露/评估要求} |
```

### 执行方式

```
如果 persona_mode = A 或 B（有操作权限的模式）:
  → 专家 Persona 实际操作 App 中的对应模块
  → 按工作流场景逐个执行，记录每步的实际体验
  → 然后完成评审任务

如果 persona_mode = C（评审模式）:
  → 专家 Persona 基于 exploration-reports 和截图完成评审
  → 不实际操作 App
```

## 6.5.5 工作流场景库：三层构建机制

> **核心认知**：专家评审质量的决定性因素不是 Persona 身份的深度，
> 而是**工作流场景的具体性**。"懂期权"只能产出"缺 Gamma 显示"，
> 而"0DTE 交易者每天 9:20 扫描 Vol Ratio 异常标的"才能产出
> "应该增加 0DTE 异动榜单 + Smart Limit 订单"。
>
> 场景越具体 → 产品缺口越清晰 → 建议越有行动力。

### 第一层：预置骨架（Preset Skeleton）

每个高频产品类别预置 5-8 个核心工作流场景。这些是经过验证的高质量场景，
作为 LLM 扩展的起点和质量标杆。

#### 金融/投资 — 期权交易专家

**场景 1: 0DTE（末日期权）日内交易**
工作流：每天 9:20 开始扫描当天到期期权 → 找 Call/Put Volume Ratio 异常（>3x）
的标的 → 检查 IV 是否飙升 → 定位 ATM 附近流动性最好的合约 → 快速下 Limit Order →
盘中根据走势 5 分钟内平仓或止损。
关键需求：0DTE 异动榜单 / 高赔率专区（期权 <$1 + 标的高波动）/ ATM 一键快捷 /
Smart Limit 订单（默认 (Ask+Mid)/2）/ 金额下单模式 / 日内机会"三张卡"降维展示

**场景 2: Earnings Play（财报博弈）**
工作流：财报前 1-3 天建仓 → 看 IV 与历史 IV 对比（判断 IV 是否被高估）→
选策略（Straddle 博大波动 / Butterfly 博窄幅）→ 财报后快速平仓。
关键需求：财报日历与期权到期日关联视图 / 历史 IV vs 当前 IV 对比图 /
Expected Move（基于 straddle price）/ 历史 Earnings Beat/Miss 与股价反应

**场景 3: 收入型策略（Covered Call / Cash-Secured Put）**
工作流：持有 100 股 → 每周/月卖出 Covered Call 收取权利金 →
选 30-45DTE、0.20-0.30 delta 合约 → 监控 delta 变化 → 必要时 roll forward。
关键需求：按 Delta 范围筛选 strike / Covered Call 一键构建（匹配持仓）/
Roll forward 一键操作 / 持仓 Greeks 汇总（portfolio-level）

**场景 4: 风险对冲（Portfolio Hedge with Puts）**
工作流：持有大量多头仓位 → 市场不确定性上升 → 买入 index puts 或个股 protective puts →
需要评估 hedge ratio 和成本。
关键需求：Portfolio beta 计算 / Hedge cost estimator / Put spread 对比工具

#### 金融/投资 — 日内交易专家

**场景 1: 盘前扫描（Pre-market Scanner）**
工作流：8:00-9:30 扫描 pre-market movers → 看 gap up/down 标的 →
检查 Level 2 和 float → 制定交易计划 → 9:30 开盘执行。
关键需求：盘前 Gap 扫描器 / Float/Short Interest / Level 2 盘前数据 / 快速 Alert

**场景 2: 突破交易（Breakout Trading）**
工作流：画支撑/阻力位 → 设条件单（突破阻力位时自动买入）→ 设止损止盈 → 监控。
关键需求：图表直接下单（chart trading）/ 条件单 / Bracket Order / 快捷键

**场景 3: VWAP 回归交易（Mean Reversion）**
工作流：盘中找偏离 VWAP 过远的标的 → 等待回归信号 → 反向开仓 → 回到 VWAP 平仓。
关键需求：VWAP 偏离率指标 / 实时 VWAP 在图表上 / 快速下单

#### 金融/投资 — 价值投资专家

**场景 1: 财报季深度研究**
工作流：财报发布 → 读 earnings transcript → 对比 guidance vs consensus →
更新 DCF 模型 → 决定加仓/减仓/持有。
关键需求：Earnings transcript 直链 / 历史 EPS 对比 / 估值指标趋势图

**场景 2: 行业横向对比选股**
工作流：选定赛道（如 Cloud SaaS）→ 拉出 10-15 家公司的关键指标 →
按 Revenue Growth + Rule of 40 + NRR 排序 → 选出 3-5 个标的深入研究。
关键需求：Peer comparison 工具 / 可自定义指标列 / 批量加入 Watchlist

（其他领域的预置骨架在 experience bank 中积累，首次使用时由 LLM 基于
appendix-domain-experts.md 的领域知识生成。）

### 第二层：LLM 动态扩展（Dynamic Expansion）

基于第一层骨架 + 实际探索数据，LLM 生成该产品特有的扩展场景。

```
扩展流程：

1. 读取产品的 exploration-reports/，提取该模块的完整功能清单
2. 对照预置场景骨架，识别产品特有的功能/入口/数据
3. 基于产品特有功能，生成 3-5 个扩展场景：

   扩展场景的生成逻辑：
   a. 产品有但骨架没覆盖的功能 → 为这个功能设计使用场景
      如: moomoo 有 "投资主题" 功能 → 生成 "主题投资追踪" 场景
   b. 产品的目标用户群中骨架未覆盖的角色 → 为这个角色设计场景
      如: 加拿大市场 → 生成 "TFSA/RRSP 税务优化" 场景
   c. 产品的竞争定位暗示的使用场景 → 设计差异化场景
      如: moomoo 强调 "0 FX fee" → 生成 "加美跨市场套利" 场景

4. 每个扩展场景必须包含：
   - 工作流（具体到时间、步骤、决策点）
   - 关键需求（3-5 个具体功能点）
   - 当前产品的支持程度（基于探索数据判断）
```

### 第三层：用户校准（User Calibration）

在专家评审执行前，将完整场景列表展示给用户确认。

```
用 AskUserQuestion 展示：

"以下是为 {product_name} 的 {module_name} 模块准备的专家评审场景：

预置场景（来自场景库）：
  1. 0DTE 日内交易 — 扫描异动 → ATM 快速下单 → 5分钟止损
  2. Earnings Play — 财报前建仓 → IV 对比 → 策略选择
  3. 收入型策略 — Covered Call 周卖 → Delta 筛选 → Roll forward
  ...

扩展场景（基于产品特点生成）：
  4. TFSA 期权税务优化 — 加拿大特有场景
  5. 投资主题关联期权 — moomoo 特有功能
  ...

你可以：
  A. 确认这些场景，开始评审
  B. 删除不相关的场景
  C. 添加你关心的具体场景（如 '跨市场期权套利'）
  D. 修改某个场景的工作流细节"
```

用户校准的核心价值：
- 用户可能有预置库中没有的独特场景（如你的 0DTE "三张卡" 概念）
- 用户可以剔除与当前评审目标无关的场景（如只关心期权，不需要价值投资场景）
- 用户输入的场景是最高质量的——因为是真实需求

### 场景沉淀到 Experience Bank

评审完成后，高价值场景自动沉淀：

```
沉淀条件：
1. 用户手动添加的场景 → 无条件沉淀（置信度 0.9）
```

## 与其他 Step 的关系

```
Step 5 (探索) → Step 5.5 (用户画像) → Step 6 (Persona 模拟)
                                            ↓
                                    Step 6.5 (专家评审) ← 可独立于 Step 6 执行
                                            ↓
                                    Step 7 (报告生成)
```

- 专家评审可以在 Step 6 之后执行（补充 Persona 反馈的深度）
- 也可以独立于 Step 6 执行（如 manual 模式不需要 Persona，但可以加专家评审）
- 专家评审的输出是独立的 `expert-review.md`，不合并到其他报告中
- `suggestion` 模式的 `optimization.md` 可以引用专家评审结果来增强建议深度
