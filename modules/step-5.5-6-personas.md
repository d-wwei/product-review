# Step 5.5 + Step 6: 用户画像分析 & 拟人用户体验模拟

> 本文件由 SKILL.md 按需加载。当子命令为 suggestion 或 ue-report 时加载。

## Step 5.5: 产品用户画像分析

**输入**：exploration-reports/ + 截图 + 产品结构大纲 + Experience Bank（如有）
**输出**：`{BASE_DIR}/persona-profiles.md`

### 5.5.0 经验检索

首次 review 时跳过。如果 `reviews/_experience_bank/` 已存在：
1. 推断产品类别 → 加载 `categories/{category}.md` + `general.md`
2. 加载 `personas/effective-combinations.md` + `anti-patterns.md`
3. 过滤置信度 >= 0.5 的条目，注入后续分析 context

### 5.5.1 产品类别与 Persona 维度识别

| 产品类别 | 关键差异化维度 | 领域专家原型（见 appendix-domain-experts.md） |
|----------|-------------|-------------------------------------------|
| 金融/投资 | 投资经验、风险偏好、资产规模、交易频率 | 期权交易者、基本面投资者、日内交易者、宏观/ETF、定投/被动、新手小白 |
| 社交/内容 | 创作能力、社交活跃度、内容偏好、隐私敏感度 | 短视频创作者、社区运营者、纯消费用户 |
| 工具/效率 | 技术水平、使用频率、工作流复杂度、付费意愿 | 自动化达人、团队管理者、轻度新用户 |
| 电商/消费 | 消费习惯、价格敏感度、品牌忠诚度 | 比价达人、跨境购物者、首次网购用户 |
| 教育/学习 | 学习目标、自律程度、时间预算、基础水平 | 备考刷题者、课程设计者、自学编程者 |

**规则**：至少 50% persona 从领域专家原型选取。涉及专业功能的原型是必选。
跨多类别产品合并维度。

### 5.5.2 用户分群矩阵

**通用维度**（所有产品必须覆盖）：年龄段(18-25/26-35/36-50/50+)、技术水平(低/中/高)、领域经验(小白/入门/进阶/专家)、使用动机(刚需/好奇/被推荐/竞品迁移)、使用场景(通勤碎片/办公专注/睡前放松)

加上产品专属维度（基于 5.5.1 的关键差异化维度）。

### 5.5.3 正交采样规则

1. 任意两个 persona 至少 3 个核心维度不同
2. 技术水平必须覆盖最低和最高各 1 个
3. 年龄段至少覆盖 3 个区间
4. 动机类型不超过 50% 相同
5. 评价倾向必须有"严格"和"包容"各 1 个

生成后输出属性矩阵表验证。不满足则调整。

### 5.5.4 Persona 数量

- 1-2 个核心功能 → 3-4 个
- 3-5 个功能模块 → 5-6 个
- 6+ 功能模块 → 7-8 个
- 平台级产品 → 8-12 个

用户通过 `--personas=N` 指定时以用户为准。

### 5.5.5 内容产品 Persona 增强（CONTENT_REVIEW 模式时激活）

对每个 Persona 追加内容消费特征：阅读习惯 + 兴趣偏好 + 内容格式偏好 + 互动倾向。

追加正交约束：至少 3 种阅读习惯、至少 1 个冷启动新用户、深度阅读者和碎片浏览者必须同时存在。

每个 Persona 模拟后额外完成：Feed 前 20 条打分(1-5)、问题内容标注(标题党/低质/重复/过期/软广)、内容满意度(1-10)、发现效率(1-5)。

## Step 6: 拟人用户体验模拟

### 6.0 Persona 档案生成（精简模板）

每个 persona 用以下精简格式，**控制在 15 行以内**：

```markdown
## Persona #{N}: {姓名}（{一句话标签}）

- **身份**: {age}岁 {profession}，{city}
- **技术水平**: {1-5} — {一句话描述}
- **领域经验**: {小白/入门/进阶/专家} — {一句话描述}
- **核心动机**: {具体场景，不要泛泛}
- **竞品经历**: {2-3个App名+使用深度}
- **行为特征**: 耐心{低/中/高} | 探索{引导依赖/自主/搜索优先} | 注意力{扫描/细读/目标导向}
- **评价风格**: {严格犀利/公正中立/包容温和}，关注{功能/视觉/效率/性价比}
- **领域专家原型**: {对应 appendix-domain-experts.md 的原型名，或"通用用户"}
```

**完整字段按需加载**：背景故事(200字)、教育/收入/家庭、评价倾向参数(5 floats)、行为指令集、情绪触发器等丰富字段存放在独立数据文件 `modules/persona-templates.md`。
- **Step 5.5 分析阶段**：只用上方精简模板（控制主 agent context）
- **Step 6.0 生成完整档案时**：Read `persona-templates.md` 获取完整字段结构，生成沉浸式角色
- **模式 A 自主操控**：完整档案 + 行为指令集注入 subagent（角色扮演需要背景故事和情绪触发器）
- **模式 B/C 评审**：精简档案 + 动机描述即可（评审不需要行为指令）

**领域知识不内联**：领域专家 persona 需要时，从 `appendix-domain-experts.md` 按需 Read 对应原型段落注入 subagent prompt。主文件只存引用。

**多样性验证矩阵**（生成后输出）：

| Persona | 年龄段 | 技术 | 领域 | 动机 | 耐心 | 评价 |
|---------|--------|------|------|------|------|------|
| 验证：任意两行≥3列不同，技术有低+高，年龄≥3区间，动机≤50%同，评价有严格+包容 |

### 6.1 行为指令（精简版）

**不再为每个 persona 生成独立的行为指令集文件。** 行为规则直接从档案属性推导：

```
行为推导规则（主 agent 或 subagent 自动应用）：
- 耐心{低} → 3秒无响应就返回，只看首屏，遇到付费立即关闭
- 耐心{高} → 愿意等待，会滚动到底，愿意填长表单
- 探索{引导依赖} → 按引导走，不主动点不熟悉的入口
- 探索{自主} → 到处点，包括设置和隐藏入口
- 探索{搜索优先} → 第一个操作是找搜索框
- 注意力{扫描} → 只看标题和大按钮，跳过说明文字
- 注意力{细读} → 每段文字都看，包括免责声明
- 注意力{目标导向} → 直奔目标功能，忽略不相关内容
```

### 反馈质量规则（所有 persona 必须遵守）

**具体性检验**：每条反馈必须能还原出 你在哪 + 做了什么 + 看到什么 + 与预期差距 + 与竞品对比。

不合格：纯情绪无事实（"界面好看"）、无量化（"加载还行"）、无上下文（"搜索不错"）。
合格：有操作（"搜'AAPL'，0.5秒出结果"）、有对比（"比 Webull 慢一倍"）、有缺失（"找不到 IV Rank"）。

**真实性规则**：Persona 是模拟画像，不是真人。禁止虚构具体时间/次数/百分比、直接引语、聚合统计。推断用"可能/预计/倾向于"。所有负面评价必须关联到 exploration-reports/ 中的具体发现。

### 6.2 三模式体验执行

#### 模式 A: 全自主体验

每个 persona 作为独立 subagent 操控 App。

**Subagent prompt 结构**（精简，控制在 80 行以内）：

```
你是 {name}，{age}岁{profession}。{motivation}。

## 你的特征
{persona 档案全文，~15行}

## 领域专业知识（仅领域专家）
{从 appendix-domain-experts.md 读取的对应原型段落，或"通用用户，无专业知识注入"}

## 行为规则
按你的耐心({patience})、探索风格({exploration})、注意力模式({attention})行动。
你不是测试员，是真正的用户。按直觉使用，在"够了"时停下。

## 反馈规则
- 每条反馈：你在哪 + 做了什么 + 看到什么 + 预期差距 + 竞品对比
- 禁止空洞评价。至少 3 处有量化数据
- 领域专家：用专业术语，覆盖测试清单 ≥70%

## 操作方法
{mobile-mcp 工具速查，6行}

## 输出
### {name} 的使用日记
- 第一印象（30秒）
- 使用过程表格（操作 | 想法 | 情绪 | 截图）
- 目标完成度
- 最喜欢/最讨厌/最困惑 各1-2条
- 领域专项测试结果表格（仅专家）
- 满意度(1-10) + 是否继续用 + 一句话总结
```

**探索深度**：耐心低→2-5分钟，中→5-15分钟，高→15-30分钟，极高→30+分钟。

#### 模式 B: 混合体验（默认）

选 3 个核心 persona 按模式 A 操控，其余按模式 C 评审。

选择规则（前 3-4 个）：
1. **新手**（技术最低+领域最低）— 测小白友好度
2. **目标用户**（最匹配核心画像）— 测核心价值
3. **领域专家**（如有）— 测专业功能深度
4. **挑剔用户**（评价最严格+竞品经验最丰富）— 测竞争力

领域专家优先：涉及专业功能时，领域专家替换挑剔用户。

#### 模式 C: 全评审体验

所有 persona 基于 exploration-reports/ 和截图评审，不操控 App。

**Subagent prompt 结构**（与模式 A 共用身份/行为/反馈规则，替换操作段）：

```
{身份+特征+领域知识+行为规则+反馈规则 — 同模式 A}

## 产品信息
{exploration_log_content}
{按模块组织的截图路径，不内联图片}

## 你的任务
有人给你展示了 {app_name} 的截图和功能介绍。以你的身份给出真实看法。
不是在写评测，是跟朋友聊天时说起这个 App。

## 评审维度
1. 第一印象
2. 功能匹配度（表格：功能|重要性|原因|竞品对比）
3. 易用性预判（一看就会/需要教/太复杂放弃）
4. 竞品对比（好于/不如/差不多）
5. 领域专项评审（仅专家：测试项|观察|专业评价）
6. 满意度(1-10) + 是否继续用 + 一句话
```

### 6.3 Persona 反馈聚合

**输出**：`{BASE_DIR}/persona-feedback-synthesis.md`

聚合结构：

```markdown
# {App} Persona 反馈综合分析

## 参与 Persona 概览
| # | 姓名 | 标签 | 体验模式 | 满意度 |

## 一、共识发现
### 共同亮点（≥50% persona 提到）
### 共同问题（≥50% persona 提到，含严重度）

## 二、分群差异
- 新手 vs 专家
- 年轻 vs 年长
- 刚需 vs 好奇

## 三、情绪热力图
| 功能/页面 | 正面 | 中性 | 负面 | 净分 |

## 四、流失风险点
| 风险点 | 受影响类型 | 流失原因 | 影响面 |

## 五、原声集锦
最正面 / 最负面 / 最有洞察 各 1 条
```
