## Step 5: 自主探索阶段

### 探索策略：先浅后深，按复杂度分组

**主 agent 职责：**
1. **浅扫**：梳理产品整体结构，每个 Tab 滚动到底，收集全部子入口
2. **估算**：对每个子入口估算复杂度和工具调用量
3. **分组**：按复杂度将队列项合并为 agent 组，非 one-page-one-agent
4. **分派**：以完成为目标分派 subagent（按组），汇总结果
5. 保持自身上下文干净

### 5.1 广度扫描（主 agent 执行）

先截图并列出当前屏幕元素：

```
mobile_take_screenshot       device: <device_id>
mobile_list_elements_on_screen  device: <device_id>
```

识别主导航结构：
- 底部 Tab Bar（iOS）/ Bottom Navigation（Android）
- 侧边栏 / 汉堡菜单
- 顶部 Tab
- 其他入口

**遍历每个主入口**，对每个入口：
1. 点击进入
2. **执行垂直滚动扫描**（至少 5 次 swipe up 或到底，产出滚动摘要。广度扫描比深度探索要求更深——遗漏一个入口意味着整个模块被跳过）
3. **水平滚动检查**：对顶部 Tab 栏/子 Tab 栏执行水平扫描：
   - list_elements 获取当前可见的 Tab 名称和数量
   - swipe left 在 Tab 栏区域，再次 list_elements 对比是否有新 Tab
   - 重复直到无新 Tab（最多 5 次水平滑动），然后滑回最左
   - 记录完整 Tab 列表（首屏 N 个 + 滑动后 M 个 = 总 N+M 个）
4. 记录该入口下的子功能列表（包括垂直滚动后和水平滑动后才可见的入口）
5. 返回主页

**主 agent 也必须遵守滚动扫描协议。** 广度扫描时很容易只看每个 Tab 的第一屏就切走——这会导致遗漏大量子入口。

```
mobile_click_on_screen_at_coordinates  device: <device_id>  x: <x>  y: <y>
mobile_take_screenshot                 device: <device_id>
mobile_list_elements_on_screen         device: <device_id>
mobile_save_screenshot                 device: <device_id>  saveTo: <path>
```

**⚠️ 广度扫描常见覆盖缺陷：**
1. **水平滚动遗漏**：实测中 Markets 子 Tab 栏首屏只显示 6 个，实际有 11 个。不执行水平滑动会漏掉 45% 模块。
2. **垂直滚动不足**：US 市场首屏 7 个模块，实际 18 个。只滚 3 次会漏掉 6-8 个模块。
3. **动态 UI 变化**：底部 Tab 栏名称可能随用户操作变化（如 AI → Papertrade）。每次返回首页时重新确认 Tab 名称。

广度扫描完成后，输出产品结构大纲到 `exploration-log.md`。

### 5.1.5 构建探索队列 + 复杂度估算 + 分组

广度扫描完成后，主 agent 在深度探索之前，必须生成一个结构化的**探索队列**并完成分组。
此队列列出所有需要深度探索的页面/入口，是后续分派 subagent 和验证覆盖完整性的核心数据结构。

#### 5.1.5a 工具调用量估算公式

对每个队列项，主 agent 根据广度扫描中的观察估算工具调用量。
**注意：估算仅用于分组决策（决定哪些页面分到一组），不作为 subagent 的执行预算。**

```
预估调用 = 导航操作(2-4)
         + 滚动扫描(每页 6-10 次: 3次swipe + 3次list_elements + 2次screenshot + 1-2次额外)
           × 页面内 Tab 数
         + 子页面递归(每个子入口 × 子页面预估调用)
         + 交互测试(每个可交互元素 3-5 次: click + screenshot + 可能的输入/list)
         + 截图(每页 3-6 张)

参考值（实测校准）:
- 单屏简单页（无子Tab，1-2个可交互元素）: ~15-25 次
- 中等页面（1-2个子Tab，3-5个可交互元素）: ~40-80 次
- 复杂页面（3+子Tab，大量可交互元素，有子页面递归）: ~100-200 次
- 超级复杂页面（8+子Tab/模块，每个Tab内还有子入口）: ~400-800 次
  示例：Markets（8个子Tab，每个Tab有列表+筛选+详情入口）、完整交易流程
```

#### 5.1.5b 细粒度分组（基于页面结构拆分）

**核心原则："宁可拆得太细，不可合得太粗"**——每组预估调用 ≤ 150 时必须拆分。
拆太细只浪费 agent 启动开销（几秒），合太粗则导致探索严重不足（整个模块遗漏）。

**分组规则（基于页面结构，不再用固定预算表）**：

| 页面类型 | 拆分规则 | 示例 |
|---------|---------|------|
| **多子Tab页面（5+子Tab）** | **每个子Tab独立1个agent** | Markets(8子Tab) → 8个agent |
| **多子Tab页面（3-4子Tab）** | **每2个子Tab合并1组** | Stock Detail(4子Tab) → 2个agent |
| **多子页面入口（10+子页面）** | **每~5个子页面1组** | Settings(13子页面) → 3个agent |
| **中等复杂度页面** | **2-3个合并一组** | Discover子模块、一般功能页 |
| **简单页面** | **3-5个合并一组** | 空状态页、单屏信息页、简单列表 |
| **超级复杂页面** | **内部模块级拆分** | 完整交易流程按阶段拆 |

**分组原则**：
1. **按结构拆分优先**：有子Tab/子页面的，按子Tab/子页面拆分，而非作为整体分配
2. **同 Tab 优先合并**：同一个 Tab 下的低/中复杂度页面优先分到一组（减少导航开销）
3. **导航连续性**：组内页面按导航路径排列，避免反复跳转
4. **组预估不超 150**：每组总预估调用量 > 150 时必须继续拆分
5. **高复杂度不合并**：预估 > 100 的页面始终独立或内部拆分

**实例对比**：
| 页面 | 旧方案 | 新方案 |
|------|--------|--------|
| Markets（8子Tab × 各有列表+筛选） | 1 agent（只探索了3/8 Tab） | **8 agents**（每子Tab一个，100%覆盖） |
| Stock Detail（Chart/Options/Fund/Comments） | 1 agent | **2 agents**（每2Tab一组） |
| Settings（13个子页面） | 1 agent | **3 agents**（每~5子页面一组） |
| 5个简单信息页 | 1 agent | 1 agent（不变） |

**复杂度分级规则**：
- **高**：股票详情页（多个子 Tab）、Settings（大量子页面）、Assets/Portfolio 等已知多屏内容、完整交易流程

**队列构建原则**：
1. **按导航效率排序**：同一个 Tab 下的页面连续排列（减少 Tab 切换开销）

3. **每个可点击入口都是一个队列项**——不合并、不跳过（分组在 agent 层面，不在队列层面）
4. **Tab 类页面拆分**：如果一个页面包含多个 Tab/标签，每个 Tab 作为独立队列项

### 5.2 深度探索（复杂度自适应分组架构）

**核心规则：按页面结构细粒度分组分派 subagent，以完成为目标（不限调用次数）。**

旧版 one-page-one-agent 在页面复杂度差异大时效率低下——简单页面浪费 agent 启动开销，
复杂页面又因预算限制探索不足。新架构按页面结构细粒度分组+完成驱动：

| 场景 | 旧版(one-page-one-agent) | 新版(完成驱动+细粒度分组) |
|------|------|------|
| Markets（8个子Tab） | 1 agent，预算100（只探索3/8） | 8 个 agent，每Tab一个（100%覆盖） |
| 股票详情页（4个子Tab） | 1 个 agent（常耗尽） | 2 个 agent（每2Tab一组） |
| 5 个简单页面 | 5 个 agent | 1 个 agent（组内串行） |

#### 5.2.0 完成驱动机制

**subagent 的唯一退出条件：所有分配页面都已彻底探索。不存在工具调用次数预算。**

```

- 报告已写入指定文件
```

```
## 页面内多 Tab 处理规则
如果目标页面包含多个 Tab/标签，**每个 Tab 都是独立页面，必须逐一点击并执行滚动扫描协议。**
不可只探索第一个 Tab 就返回。

## 去重规则
当你点击一个入口进入新页面时：
1. 判断该页面的唯一标识（如 stock-detail:MSFT、markets:overview）
2. 检查是否在"已探索页面"列表中：
   - 已探索 → 截图记录"此页面已探索"，立即返回
   - 未探索 → 执行完整探索
3. 对于股票列表中的个股：
   - 只需点击 1-2 只代表性个股验证详情页结构
   - 不同市场（US vs CA）至少各探索一只
```

#### 5.2.2 长页面接力机制（Relay Protocol）

当 subagent 报告 `[接力点]`（scroll_count 达到 10 且页面仍有新内容时），主 agent 派生接力 subagent：

```
接力规则：

1. 当 subagent 的滚动次数达到 10 且页面仍有新内容（非无限滚动）：
   - 当前 subagent 保存当前位置截图
   - 报告标注：[接力点] scroll_count=10, 最后可见元素={列表}
   - 结束当前 subagent（不返回首页）

2. 主 agent 派生接力 subagent，prompt 包含：
   "你是接力探索员。前一位在 {page_name} 滚动了 10 次到达 {位置描述}。
    请从当前屏幕位置继续向下滚动，直到到底。
    然后探索页面上尚未被前一位点击的入口。"

3. 接力最多 3 轮（即最深 30 次滚动 + 安全上限 15 = 45 次）

4. 接力 subagent 的报告追加到同一个报告文件中
```

#### 5.2.2.5 运行时自动拆分（Complexity Overflow Protocol）

当 subagent 在探索过程中发现实际复杂度远超预期时（如一个页面有大量未预期的子入口），
执行以下流程而非强行在安全上限内赶进度：

```
运行时自动拆分流程：

1. Subagent 发现复杂度溢出（如：分配了 3 个页面，但第 1 个页面就有 20+ 子入口）
2. Subagent 正常完成已探索的页面（不降低探索质量）
3. 写入完整报告（已探索部分）
4. 在报告末尾和返回摘要中标记未完成页面：
   状态: [实际复杂度远超预期-建议拆分]
   原因: "页面 X 有 N 个子入口，预估 ~M 次调用，远超组内预估"
   建议: "建议拆分为 K 个独立 agent"
5. 主 agent 收到标记后：
   a. 将未完成页面从原组中移出
   b. 按 5.1.5b 规则重新细粒度分组
   c. 派发新的 subagent 处理
```

**关键原则**：宁可多派一个 agent，也不要降低探索质量来"赶进度"。

#### 5.2.2.6 大型任务预判拆分（报告生成阶段适用）

对"读写密集型"任务（专家评审、说明书、优化建议），分派前必须预判规模：

| 维度 | 阈值 | 超过时处理 |
|------|------|----------|
| 输入文件总行数 | > 2000 行 | 必须拆分（每批 ≤ 3 个文件） |
| 输入文件数量 | > 6 个 | 必须拆分 |
| 预期输出长度 | > 500 行 | 必须分段写入 |
| 输入+输出总量 | > 3000 行 | 禁止单 agent，必须拆分 |

**拆分策略**：
- **专家评审**：按模块拆（每模块 1 agent，读 1-2 个报告），主 agent 合并
- **MECE 说明书**：按一级模块拆（每模块 1 agent），主 agent 合并
- **优化建议**：主 agent 在 prompt 中直接内嵌精简摘要（<4000字），不让 agent 读文件
- **通用规则**：宁可多派 2-3 个小 agent 再合并，不赌单个大 agent

#### 5.2.3 Subagent 导航效率规则

```
导航标准化规则：

1. 每个 subagent 结束时，必须将设备导航回"底部 Tab 首页"之一。
   优先回到与下一个 subagent 目标最近的 Tab。

2. 主 agent 分派时告知当前设备位置。

3. Subagent 从当前位置出发导航，不假设在首页。

4. 组内页面已按导航效率排序：
   - 同一 Tab 下的页面在同一组内连续探索
   - 同一详情页的多个子 Tab 由同一 subagent 连续探索
```

```
如果 PARALLEL_MODE=true 且 DEVICES.length > 1:
```
